# Automate GitHub Release and Discussion creation on tag push

> **Note:**
> This ADR's "ungated" decision, argued below, is superseded by [`.github` ADR 0003](https://github.com/dnd-mapp/.github/blob/main/docs/adr/0003-gate-release-creation-behind-required-reviewer-environment.md): release creation across the platform, including this repo's, is now gated behind a required-reviewer GitHub Environment. The `validate`-job shape described below is now the platform-wide convention, see [`.github` ADR 0002](https://github.com/dnd-mapp/.github/blob/main/docs/adr/0002-signed-and-validated-release-tags.md). The rest of this ADR (why release/discussion creation is automated at all, and the `package.json`-sync reasoning) is unaffected and still holds.

A [release.yml](../../.github/workflows/release.yml) workflow now runs on every `vX.Y.Z` tag push. Its `validate` job checks that the tag is stable semver, that the tagged commit is reachable from `main`, that the tag's version matches `package.json`'s `version` field, and that `CHANGELOG.md` has a matching `## [X.Y.Z] - YYYY-MM-DD` section, then re-runs `pnpm format-check`/`pnpm lint-md`, since a tag can point at any commit, not just one that already passed `pull-request.yml`/`push-main.yml`. If `validate` passes, `create-release` extracts that changelog section and runs `gh release create` with `--discussion-category "Announcements"`, which creates both the GitHub Release and a mirroring Announcements Discussion in one step. There is no build/pack/publish step: a composite action has no artifact to ship, since consumers reference this repo directly via `uses:`.

`package.json`'s `version` field is kept in sync with each release's tag and changelog entry, updated by hand in the same "prepare release" PR that renames `CHANGELOG.md`'s `[Unreleased]` heading (see [Releasing a new version](../guides/dev/releasing.md)), even though nothing in this repo consumes that field at runtime. It exists so `validate` has a single machine-checked field to compare the tag against, rather than relying solely on the changelog heading's free-form text.

This supersedes the "automating this ... was rejected for now" reasoning in [ADR 0001](0001-manual-tags-with-floating-major-version.md) for the release/discussion-creation piece specifically. ADR 0001's core decision (manual immutable tags, and a manually-moved floating `v1`) is unchanged.

## Considered options

- Gate `create-release` behind a required-reviewer GitHub Environment, like tsconfig's `package-publish`. Rejected at the time: there's no registry publish or other irreversible external action for a gate to protect here, the tag push already is the approval step. Later reversed platform-wide, see the note at the top of this ADR.
- Have `release.yml` itself bump `package.json`/`CHANGELOG.md`. Rejected for the same reason as tsconfig's [ADR 0003](https://github.com/dnd-mapp/tsconfig/blob/main/docs/adr/0003-manual-version-bumping.md): a tag-triggered workflow writing back to `main` is awkward, and this repo's changelog is already hand-maintained with no automation to build on.
- Skip `package.json` version-syncing and validate the tag against the changelog heading directly. Rejected: changelog headings are free-form prose, not a single machine-checked field, so mismatches (typos, wrong date format) would be harder to catch reliably than a straight string comparison against `package.json`.
